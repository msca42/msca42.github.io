大型网站架构演化

发表于 2019-08-31 | 更新于 2019-09-05 | 分类于 架构
  • 本文为《大型网站技术架构：核心原理与案例分析》第一章的读书笔记总结
  • 第一阶段：
    • 文件，应用，数据库
    • LAMP
    • 文件，应用程序和数据库共存于一台服务器中
  • 第二阶段：
    • 文件，应用，数据库
    • 应用服务和数据服务分离
    • 三台服务器：应用服务器，文件服务器和数据库服务器
    • 针对不同需求定制相应的服务器功能
    • 原因
      • 单一服务器不能满足文件容量，抑或是CPU处理的需求
  • 第三阶段：
    • 分布式缓存集群，文件，应用(本地缓存)，数据库
    • 缓存
    • 新增本地缓存和远程分布式缓存集群
    • 原因
      • 根据八二原则：热点数据只有百分之20，所以将其缓存
  • 第四阶段：
    • 负载均衡，应用集群(本地缓存)，文件，分布式缓存集群，数据库
    • 应用服务器集群(负载均衡器)
    • 原因
      • 单一服务器无法处理过量的请求
    • 负载均衡器可以将用户请求分发到任意服务器上
  • 第五阶段：
    • 负载均衡，应用集群(本地缓存，数据访问模块)，分布式缓存集群文件，数据库(主从热备，读写分离)
    • 数据库（主从热备，读写分离）
    • 原因：
      • 缓存之后，读操作从缓存中读取，除了部分缓存访问不命中，缓存过期；全部写操作
      • 以上操作，单一数据库不能很好的满足
  • 第六阶段
    • CDN，反向代理，负载均衡，应用集群(本地缓存，数据访问模块)，分布式缓存集群，文件，数据库(主从热备，读写分离)
    • 反向代理和CDN
      • CDN:网络提供商的机房
      • 反向代理：网站的中心机房(最先访问)
    • 原因：
      • 优化用户体验，提高网站加载速度；
      • 尽早的返回数据给用户，加快用户访问速度，减轻后端服务器的负载压力
  • 第七阶段
    • CDN，反向代理，负载均衡，应用集群(本地缓存，统一数据访问模块)，分布式缓存集群，分布式文件，分布式数据库
    • 分布式数据库，分布式文件服务器
    • 关于分布式数据库：
      • 分布式数据库只有在单表数据规模非常庞大
      • 更常用手段是业务发分库，将不同业务的数据库部署在不同的物理服务器。
    • 原因：数据规模扩大
  • 第八阶段
    • CDN，反向代理，负载均衡，应用集群(本地缓存，统一数据访问模块)，分布式缓存集群，分布式文件，分布式数据库
    • 分布式数据库，分布式文件服务器，搜索引擎，NoSQL
    • NoSQL,搜索引擎
    • 原因：
      • NoSQL,搜索对可伸缩的分布式特性具有更好的支持
      • 应用服务器用过统一数据访问模块访问各种数据，减轻应用程序管理诸多数据源的麻烦
  • 第九阶段
    • CDN，反向代理，负载均衡，应用集群A,消息队列，应用集群B(本地缓存，统一数据访问模块)，分布式缓存集群，分布式文件，分布式数据库
    • 应用集群粒度细化，消息队列
      • 将网站拆分成许多不用的应用，每个应用独立部署维护
      • 通过超链接或者通过消息队列进行数据分发
  • 第10阶段
    • CDN，反向代理，负载均衡，应用集群A,消息队列，应用集群B，分布式服务(本地缓存，统一数据访问模块)，分布式缓存集群，分布式文件，分布式数据库
    • 分布式服务
      • 将可复用的应用提取出来，应用系统只需管理用户界面，通过分布式服务调用公用业务服务
  • 参考来源：《大型网站技术架构：核心原理与案例分析》

JVM知识总结

发表于 2019-08-21 | 更新于 2019-09-24 | 分类于 JVM

前言

本文主要参考周志明的第二版《深入理解Java虚拟机》，芋道源码精尽java【虚拟机】,陈树义《Java基础系列》，对JVM知识进行一次有体系的复习。

思维导图

正文

JVM是什么？

  • Java Virtual Machine，java虚拟机，主要任务是编译字节码文件，生成机器码。
  • Sun的定义有所区别:JVM指Java HotSpot Client and Server VM和Interpreter
  • Java的平台无关性的重要体现就是JVM；因为JVM处理的对象字节码是跨平台的（write once ,run everywhere），而且JVM屏蔽了操作系统软硬件的差异，所以只要在不同操作系统上搭建对应的JVM平台就可以完成相应的编译过程。

JVM组成?

  • JVM主要由类加载器，运行时数据区，执行引擎，本地接口组成

编译与优化

JVM处理的对象是class文件，它是通过java文件javac编译而来的，然后通过JVM生成机器码，亦或是直接将java文件编译称本地机器码。上述过程涉及了java的几种编译器。

  • 前端编译器：Sun的Javac。
  • JIT编译器：HotSpot VM的C1，C2编译器。
  • AOT编译器：jaotc。
javac的编译过程

  • 解析和填充符号表过程
    • 词法分析（parseFile）
      • 将源代码的字符流转变为标记（Token）集合;
      • 单个字符式程序编写过程的最小元素，标记是编译过程的最小元素。
      • 标记范围：关键字，变量名，字面量，运算符
    • 语法分析
      • 根据Token序列构造抽象语法树（AST）
      • AST是一种用来描述程序代码语法结构的树形表示方式，每个节点都代表着程序代码中一个语法结构
    • 输入到符号表(enterTrees)
      • 符号表是由一组符号地址和符号信息构成的表格。
      • 如果用户代码中没有提供任何构造器，填充时自动加入默认构造函数
      • 符号表的作用
        • 语义分析:语义检查
        • 目标代码生成阶段，对符号名进行地址分配，符号表是地址分配依据
  • 插入式注解处理器的注解处理过程
  • 语义分析与字节码生成过程
    • 标注检查(attribute)
      • 变量使用前是否已声明
      • 变量与赋值之间的数据类型是否匹配
      • 常量折叠
        • 常量优化机制
        • 举例说明：语法树 int a=1+2;编译期间 int a=3;
    • 数据流分析(flow)
      • 对程序上下文逻辑更进一步的验证
      • 程序局部变量在使用前是否由赋值
      • 方法的每条路径是否都有返回值
      • 是否所有的受查异常都被正确处理
    • 解语法糖（desugar）
      • 将java语法糖还原回简单的基础语法结构
    • 生成字节码
      • 将前面生成的信息转化称字节码写到磁盘中
      • 添加实例构造器init()和类构造器clinit()到语法树中
      • 这里生成构造器的过程是一个收敛的过程：
        • 先执行父类的实例构造器，然后初始化变量，最终执行语句块
      • 优化：字符串的加操作替换为StringBuffer或StringBuilder（jdk>1.5）的append()
      • 由ClassWriter类的writeClass()方法输出字节码，生成最终的Class文件
运行期优化

解释器和编译器

  • HotSpot同时包含解释器（interpreter）和编译器(c1,c2)
    采用混合模式解释器+一个编译器
  • 解释器
    • 程序迅速启动和执行的时候
    • 内存资源限制,采用解释器节约内存
    • 作为逃生门，逆优化，提高效率
  • 编译器（即时编译）
    • 将代码编译成本地代码，提高执行效率
    • C1:Client Compiler
    • C2:Server Compiler

分层策略

  • 第0层:程序解释执行，解释器不开启性能监控（Profiling）,可触发第一层编译
  • 第1层：C1编译，将字节码编译为本地代码，进行简单，可靠的优化，如有必要将加入新跟那个监控的逻辑
  • 第2层(及以上)：C2编译，将字节码编译为本地代码，但是会启用一些编译耗时较长的优化，甚至会根据性能监控信息进行一些不可靠的激进优化
  • 分层之后，C1,C2同时工作
    • C1获取更高的编译速度
    • C2获取更好的编译质量
    • 解释执行不需要收集性能监控信息

热点代码

  • 热点探测（Hot Spot Detection）
    • 基于采样的热点探测
      • 周期性检查线程的栈顶，经常出现即可
      • 优点：实现简单，高效，很容易地获取方法调用关系
      • 缺点：易受线程阻塞
    • 基于计数器的热点探测 （HotSpot）
      • 为每个方法（甚至代码块）建立计数器
  • 被多次调用的方法
    • 以整个方法作为编译对象，标准的JIT编译方式
  • 被多次执行的循环体
    • 以整个方法为编译对象，栈上替换(OSR编译)

HotSpot为每个方法提供了两个计数器：

  • 方法调用计数器（Invocation Counter）
  • 回边(Back Edge Counter)
  • 超过阈值，触发JIT
    • 方法调用计数器 Client:1500 Server:10000
    • 一个方法被调用
      • 该方法是否存在被JIT编译过的版本
      • 存在，优先使用本地代码
      • 不存在，此方法的调用计数器值加1，判断方法调用计数器+回边计数器>方法调用阈值，向JIT提交该方法的代码编译请求
      • 执行引擎继续进入解释器，循环之前的流程，直到编译完成。
      • 如果一段时间，方法调用计数未满足要求，进入半衰期，热度衰减
    • 回边计数器
      • 统计一个方法中循环体代码的执行的次数
      • 回边：在字节码中遇到控制流向后跳转的指令
      • 目的：触发OSR编译

原理探索

xml文档解析

  • 这里的xml遵守Schema约束。以下是逐行分析

  • 第一行

说明这是一个xml文件，相当于告诉解析器这是个XML文档，你需要用XML解析器解析

  • 第二行

    xmlns ： XML Namespaces （命名空间）；

    beans 是xml的根节点

    所在行的命名空间，是一种默认命名空间（只能存在一个）。没有使用命名空间前缀，表示未使用其他命名空间的所有标签的默认命名空间，相当于一个兜底。

  • 第三行

    w3c 规范， xsi 对应到XMLSchema-instance

  • 第四行

    具体用到的资源，用于声明目标名称空间的模式文档。

    xsi:schemaLocation属性的值由多个URI引用组成，两个URI之间以空白符分隔。第一个URI是命名空间的名字，第二个URI给出模式文档的位置，模式处理器将从这个位置读取模型文档，该模式文档的目标命名空间必须与第一个URI像匹配

XML文档加载顺序

Spring 在加载xsd文件时总是先试图在本地查找xsd文件(Spring的jar包中已经包含了所有版本的xsd文件)，如果没有找到，才会转向URL指定的路径下载。 Spring加载xsd文件的类是PluggableSchemaResolver,或者通过日志

了解spring 是如何加载xsd文件。此外，Spring 在本地找不到需要的文件向网站下载原因： Spring jar 包的版本低于xsd的文件版本，如果此时发生了断网问题，就会出现问题。

总结（个人意见）

Spring主要是对bean(对象)的处理，所以在配置文件中把beans作为默认命名空间。又因为Schema约束（w3），所以xmln需要与url对应。问题就出在是否遵守xml文件的约束和解析条件，以便xml文件可以找到http://www.springframework.org/schema/beans 与http://www.springframework.org/schema/beans/spring-beans.xsd。

第一章：计算机系统漫游（读书笔记）

发表于 2019-04-08 | 更新于 2019-09-05 | 分类于 深入理解计算机系统

​

CSAPP主要讲了什么？

​ 计算机种的某些软硬组件是如何工作以及如何影响程序正确性和性能的。

综合论述

​ 第一章主要讲了两个大方面的内容，一个是数据存储的硬件流程，一个是hello.c的编译流程；

​ 所谓信息，就是计算机中存储的数据；在计算机种，数据的存储是以一串比特（字节）序列为媒介。而如何区分数据

对象就要依靠上下文。

​ 所谓上下文，就是OS保持跟踪进程运行所需的所有状态信息；比如PC和寄存器的当前值，以及主存的内容。这些状

态信息在计算机中伴随着进程的运行动态变化；

​ PC存放了下一条指令的内存单元地址，CPU每执行一条命令，PC便转向下一条指令；执行命令时，寄存器依据

CPU，从主存中复制信息到寄存器，进行覆盖；抑或将本地数据存储到主存当中。如果要进行算术运算，CPU便会把两个

寄存器中的值交给ALU（算术逻辑单元）进行计算，并返回给其中一个寄存器。

​ CPU执行进程并非单一的，而是抽象并行的，在某一个程序运行过程中，CPU可能会交替运行另一个程序，在这之

前，CPU会记录当前的上下文状态，以便复原。这样的行为叫做上下文切换（CPU切换）。

​ CPU和内存之间的交换信息，是通过总线来进行的，CPU通过系统总线连接北桥，北桥通过内存总线与主存相连；

内存是DRAM芯片组成，每个字节都有自己唯一的地址。

​ 内存除了和CPU交换信息之外，和硬盘以及其他的I/O设备都进行数据交换；

​ 虚拟内存：

​ 程序代码和数据： 加载进内存的可执行文件代码

​ 堆（运行时堆）：当调用像malloc和free这样的C标准库时，堆可以在运行时动态地扩展和收缩

​ 共享库： 存放C标准库和数学库等共享库的代码和数据。

​ 栈： 我们进行函数调用和函数退出的时候，栈空间进行扩展和收缩。

​ 内核虚拟内存：不允许应用程序进行着区域的内容读写。应用程序必须调用内核提供的接口来进行操作。

​ Hello.c 编译的四个阶段

1. 预处理 ：导入系统头文件，形成hello.i
2. 编译： 翻译成汇编语言 ，形成hello.s
3. 汇编：将汇编语言翻译成机器语言，形成hello.o的二进制文件
4. 链接:  将printf.o 合并到源文件中，形成可执行目标文件 ，加载到内存，由系统执行

2. 比较下C语言的链接和Java 链接有什么不同？

​    C: 

​        进过编译后，生成一个目标文件，这个文件调用printf函数，对于printf文件，目标代码在/usr/lib中，链接所

​    要做的就是将这些函数库中的代码组合到目标文件中。

​    Java：  

​       将java类中的二进制代码合并到JVM的运行状态之中的过程 。在链接之前，这个类必须被成功加载。

​       类的链接包括验证，准备和解析；

​            验证： 确保java类的二进制表示在结构上完全正确；异常:java.lang.VerifyError

​            准备： 创建java类中的静态域，并将这些域的值设为默认值

​            解析：确保引用类能被正确找到-->其他java类被加载



3. 什么是RISC指令集，什么是CISC指令集？

​    RISC：精简指令集

​        特点： 微指令集较为精简，每个指令的执行时间都很短，完成的动作单纯，指令的执行性能较佳；但要做

​    复杂的事情，就要由多个指令来完成

​    CISC 复杂指令集

​        特点：每个小指令可以执行一些较低阶的硬件操作，指令数目多而且复杂，每条指令的长度并不相同。因为

指令执行较为复杂，所以每条指令花费的时间较长，但每条个别指令可以处理的工作较为丰富。



4. 什么是基于栈的指令集和基于寄存器的指令集？

​    1.基于栈的指令集

​        优点：可移植（寄存器由硬件提供，程序依赖硬件寄存器会收到硬件约束），基于栈的指令集架构，用户不

会直接使用这些寄存器，由虚拟机实现决定把一些访问频繁的数据放到寄存器中获得尽量更好的性能；还有优点是代码

相对紧凑（字节码中每个字节对应一条指令）、编译器实现简单（无需考虑空间分配）等；

​        缺点：执行速度慢，效率较低。因为操作本身伴随着大量的入栈出栈，导致完成相同功能所需的指令数量增加；

更重要的是栈实现在内存中，频繁的栈访问意味着频繁的内存访问，这大大降低了效率。

​    2.基于寄存器的指令集

​        优点：效率相对较高

​        缺点：平台依赖，可移植性差。编译实现复杂。

6. 南桥和北桥在什么位置？ 有什么作用？

​ 北桥： 负责连接速度较快的CPU，内存与显卡接口等元件—>CPU和内存之间的桥接–.>大多将北桥内存控制器整合到CPU封装

​ 南桥： 负责连接速度较慢的设备接口,在主板上。
