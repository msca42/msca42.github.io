使用环境

  • springframework源码中自己新建的模块，并在build.gradle中添加对spring模块的依赖

    compile(project(“:spring-beans”))

案例代码

MyTestBean

BeanFactoryTest

错误提示

必须具有偶数个 URI

问题分析

根据错误提示，现在只有奇数个URI，即xsi：schemaLocation 前后不匹配

经过对比，发现书中案例的XML标签存在大小写。将Spring改成spring 即可。

原理探索

xml文档解析

  • 这里的xml遵守Schema约束。以下是逐行分析

  • 第一行

说明这是一个xml文件，相当于告诉解析器这是个XML文档，你需要用XML解析器解析

  • 第二行

    xmlns ： XML Namespaces （命名空间）；

    beans 是xml的根节点

    所在行的命名空间，是一种默认命名空间（只能存在一个）。没有使用命名空间前缀，表示未使用其他命名空间的所有标签的默认命名空间，相当于一个兜底。

  • 第三行

    w3c 规范， xsi 对应到XMLSchema-instance

  • 第四行

    具体用到的资源，用于声明目标名称空间的模式文档。

    xsi:schemaLocation属性的值由多个URI引用组成，两个URI之间以空白符分隔。第一个URI是命名空间的名字，第二个URI给出模式文档的位置，模式处理器将从这个位置读取模型文档，该模式文档的目标命名空间必须与第一个URI像匹配

XML文档加载顺序

Spring 在加载xsd文件时总是先试图在本地查找xsd文件(Spring的jar包中已经包含了所有版本的xsd文件)，如果没有找到，才会转向URL指定的路径下载。 Spring加载xsd文件的类是PluggableSchemaResolver,或者通过日志

了解spring 是如何加载xsd文件。此外，Spring 在本地找不到需要的文件向网站下载原因： Spring jar 包的版本低于xsd的文件版本，如果此时发生了断网问题，就会出现问题。

总结（个人意见）

Spring主要是对bean(对象)的处理，所以在配置文件中把beans作为默认命名空间。又因为Schema约束（w3），所以xmln需要与url对应。问题就出在是否遵守xml文件的约束和解析条件，以便xml文件可以找到http://www.springframework.org/schema/beans 与http://www.springframework.org/schema/beans/spring-beans.xsd。

跳出xml文件规则本身，Spring 将xml解析为bean的过程是怎样？头文件在其中起到什么样的作用？

第一章：计算机系统漫游（读书笔记）

发表于 2019-04-08 | 更新于 2019-04-14 | 分类于 深入理解计算机系统

​

CSAPP主要讲了什么？

​ 计算机种的某些软硬组件是如何工作以及如何影响程序正确性和性能的。

综合论述

​ 第一章主要讲了两个大方面的内容，一个是数据存储的硬件流程，一个是hello.c的编译流程；

​ 所谓信息，就是计算机中存储的数据；在计算机种，数据的存储是以一串比特（字节）序列为媒介。而如何区分数据

对象就要依靠上下文。

​ 所谓上下文，就是OS保持跟踪进程运行所需的所有状态信息；比如PC和寄存器的当前值，以及主存的内容。这些状

态信息在计算机中伴随着进程的运行动态变化；

​ PC存放了下一条指令的内存单元地址，CPU每执行一条命令，PC便转向下一条指令；执行命令时，寄存器依据

CPU，从主存中复制信息到寄存器，进行覆盖；抑或将本地数据存储到主存当中。如果要进行算术运算，CPU便会把两个

寄存器中的值交给ALU（算术逻辑单元）进行计算，并返回给其中一个寄存器。

​ CPU执行进程并非单一的，而是抽象并行的，在某一个程序运行过程中，CPU可能会交替运行另一个程序，在这之

前，CPU会记录当前的上下文状态，以便复原。这样的行为叫做上下文切换（CPU切换）。

​ CPU和内存之间的交换信息，是通过总线来进行的，CPU通过系统总线连接北桥，北桥通过内存总线与主存相连；

内存是DRAM芯片组成，每个字节都有自己唯一的地址。

​ 内存除了和CPU交换信息之外，和硬盘以及其他的I/O设备都进行数据交换；

​ 虚拟内存：

​ 程序代码和数据： 加载进内存的可执行文件代码

​ 堆（运行时堆）：当调用像malloc和free这样的C标准库时，堆可以在运行时动态地扩展和收缩

​ 共享库： 存放C标准库和数学库等共享库的代码和数据。

​ 栈： 我们进行函数调用和函数退出的时候，栈空间进行扩展和收缩。

​ 内核虚拟内存：不允许应用程序进行着区域的内容读写。应用程序必须调用内核提供的接口来进行操作。

​ Hello.c 编译的四个阶段

1. 预处理 ：导入系统头文件，形成hello.i
2. 编译： 翻译成汇编语言 ，形成hello.s
3. 汇编：将汇编语言翻译成机器语言，形成hello.o的二进制文件
4. 链接:  将printf.o 合并到源文件中，形成可执行目标文件 ，加载到内存，由系统执行

2. 比较下C语言的链接和Java 链接有什么不同？

​    C: 

​        进过编译后，生成一个目标文件，这个文件调用printf函数，对于printf文件，目标代码在/usr/lib中，链接所

​    要做的就是将这些函数库中的代码组合到目标文件中。

​    Java：  

​       将java类中的二进制代码合并到JVM的运行状态之中的过程 。在链接之前，这个类必须被成功加载。

​       类的链接包括验证，准备和解析；

​            验证： 确保java类的二进制表示在结构上完全正确；异常:java.lang.VerifyError

​            准备： 创建java类中的静态域，并将这些域的值设为默认值

​            解析：确保引用类能被正确找到-->其他java类被加载



3. 什么是RISC指令集，什么是CISC指令集？

​    RISC：精简指令集

​        特点： 微指令集较为精简，每个指令的执行时间都很短，完成的动作单纯，指令的执行性能较佳；但要做

​    复杂的事情，就要由多个指令来完成

​    CISC 复杂指令集

​        特点：每个小指令可以执行一些较低阶的硬件操作，指令数目多而且复杂，每条指令的长度并不相同。因为

指令执行较为复杂，所以每条指令花费的时间较长，但每条个别指令可以处理的工作较为丰富。



4. 什么是基于栈的指令集和基于寄存器的指令集？

​    1.基于栈的指令集

​        优点：可移植（寄存器由硬件提供，程序依赖硬件寄存器会收到硬件约束），基于栈的指令集架构，用户不

会直接使用这些寄存器，由虚拟机实现决定把一些访问频繁的数据放到寄存器中获得尽量更好的性能；还有优点是代码

相对紧凑（字节码中每个字节对应一条指令）、编译器实现简单（无需考虑空间分配）等；

​        缺点：执行速度慢，效率较低。因为操作本身伴随着大量的入栈出栈，导致完成相同功能所需的指令数量增加；

更重要的是栈实现在内存中，频繁的栈访问意味着频繁的内存访问，这大大降低了效率。

​    2.基于寄存器的指令集

​        优点：效率相对较高

​        缺点：平台依赖，可移植性差。编译实现复杂。　



5. 在进程的虚拟地址空间中， 有用户栈，运行时堆等区域， 如果一个进程运行了多个线程，用户栈是怎么使用的？
6. 南桥和北桥在什么位置？ 有什么作用？

​ 北桥： 负责连接速度较快的CPU，内存与显卡接口等元件—>CPU和内存之间的桥接–.>大多将北桥内存控制器整合到CPU封装

​ 南桥： 负责连接速度较慢的设备接口,在主板上。
