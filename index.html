层次与组件分析

  • Server

    • 连接器： 连接处理，授权认证，安全权限

      每个客户端连接都会在服务器进程中拥有一个线程,这个连接的查询只会在这个单独的线程中执行，该线程只能轮流在某个CPU核心或CPU中运行。服务器会负责缓存线程，因此不需要为每个新建的连接创建或销毁线程。

    • 太史曰： 上述是一种线程池的概念,所谓线程池，就是实现创建好许多线程，在每个客户端连接服务器的时候总有一个线程与之对应即可。

    • 认证

      • SSL连接采用X.509证书认证。
    • 权限检查
    • 查询缓存：缓存完整的SELECT查询结果

    • 分析器: 创建解析树，并进行词法和语法分析

并发及其相关问题

  • 并发控制： 在处理并发读或者写时，通过一个由两种类型锁组成的锁系统。
    • 读写锁
      • 读锁：互不阻塞，互不干扰，共享锁
      • 写锁：阻塞其他锁，排他锁
    • 锁粒度
      • 目的：提高并发性
      • 方式：修改锁定的数据片精度，锁定的数据量越小，并发程度越高
    • 锁策略：在锁的开销和数据的安全性之间寻找平衡，影响性能。
      • 表锁：最基本，开销最小，锁定整张表 ，写锁优先级更高
      • 行级锁：最大程度支持并发，锁开销最大

事务

  • 事务是什么？
    • 一组原子性的SQL查询，或者说一个独立的工作单元
  • ACID:
    • 原子性
      • 一个事务必须被视为一个不可分割的最小工作单元，整个事务中的所有操作要么全部提交成功，要么全部失败回滚
    • 一致性
      • 数据库总是从一个一致性状态转到另一个一致性状态
    • 隔离性
      • 一个事务所作的修改在最终提交之前，对其他事务是不可见的
    • 持久性
      • 一旦事务提交，则其所做的修改就会永久保存在数据库中

  • 隔离级别

    • 观察点：一个事务中所做的修改，在事务内部和事务间可见或不可见

    • 分类与缺陷：

      • 读未提交：事务的修改，即使没有提交，对其他事务也是可见的
        • 脏读：事务可以读取未提交的数据
      • 读提交：（大多数数据库采用） 一个事务开始时，只能看见已提交的事务所作的修改
        • 不可重复读：一个事务从开始到提交，所作的任何修改对其他事务都是不可见的，即执行两次查询操作得到的结果可能不同
      • 可重复读：保证了在同一事务中多次读取的数据一致（MySQL默认）
        • 幻读：当某个事务在读取某个范围内的记录时，另一个事务又在该范围内插入新的记录，当之前的事务再次读取该范围的记录时，会产生幻行
        • 幻读通过多版本并发控制(MVCC)解决
      • 串行化：强制事务串行执行

    • 死锁：两个或多个事务在同一资源上相互占用，并请求对方占用的资源，从而导致恶性循环的过程

    • 解决方法
      - 死锁检测：InnoDB将持有最少行级排他锁的事务进行回滚
- 死锁超时机制
    • 事务日志：
      • 存储引擎在修改表的数据时只需要修改器内存拷贝，再把该修改行记录到硬盘上的事务日志中，而不用每次都将修改的数据存储在磁盘上。
      • 事务日志采用追加写入，所以磁盘小区域顺序I/O
      • 事务日志持久后，内存中修改的数据慢慢回刷到磁盘，两次写入
    • MySQL中的事务：
      • 事务存储引擎：InnoDB,NDB Cluster
      • 自动提交：每个查询可以看作一个事务
      • InnoDB采用两阶段锁定协议:随时上锁，锁只有在COMMIT或ROLLBACK的时候才会释放。（隐式+显式）

多版本并发控制（MVCC）

  • 简介
    • 通过保存数据在某个时间点的快照来实现。
    • 行级锁的变种，典型：乐观锁，悲观锁
    • 不管需要指定多长时间，每个事务看到的数据都是一致的。根据事务开始的时间不同，每个事务对同一张表，同一时刻看到的数据可能是不一样的。
    • 只在可重复读和读已提交下工作
  • InnoDB的MVCC
    • 通过在每行记录后面保存两个隐藏的列来实现。
    • 一个保存了行的创建时间，一个保存行的过期时间（删除时间）。
    • 记录的是系统版本号，开始一个新事务，版本号递增
    • 事务开始时刻的版本号会作为事务的版本号
    • 具体操作：
      • SELECT
        • 只查找版本早于当前事务版本的数据行。
        • 行的删除版本要么未定义，要么大于当前事务版本号。
      • INSERT
        • 新插入的每一行保存当前系统版本号作为行版本号
      • DELETE
        • 删除的每一行保存当前系统版本号作为行删除标识
      • UPDATE
        • 插入新数据，保存当前系统版本号作为行版本号，同时保存当前系统版本号到原来的行作为删除版本号

MySQL存储引擎

  • 概述
    • MySQL将每个数据库（schema）保存为数据目录下的一个子目录。
    • 创建表时，会在数据库子目录下创建一个和表同名的.frm文件保存表的定义

InnoDB

  • 概述
    • 数据存储在表空间
    • 默认可重复读，通过间隙锁策略防止幻读。间隙锁不仅锁定查询涉及的行，还会对索引中的间隙进行锁定
    • InnoDB表通过聚簇索引建立
    • 支持热备份
    • 可预测性预读：能够自动在内存中创建hash索引以加速读操作的自适应哈希索引，以及能够加速插入操作的插入缓冲区

补充

一.数据库服务器与嵌入式数据库的区别？

  • 运行地址空间不同
    • 数据库服务器独立运行一个守护进程
    • 嵌入式数据库与应用程序运行在一个进程
  • 架构不同
    • 数据库服务器C/S结构，需要驱动程序，通过TCP/IP通讯
    • 嵌入式数据库直接将库文件链接到应用程序中，通过API访问数据

三. 长连接与短连接

  • 长连接
    • 连接成功后，如果客户端持续由请求，则一直使用同一个连接
    • MySQL在执行过程中临时使用的内存是管理在连接对象中，所以长连接容易OOM
  • 短连接
    • 每次执行完很少的几次查询就断开连接，下次查询再重新建立一个

四.InnoDB表空间

  • 共享表空间
    • 整个数据库的表数据和索引存储在一个文件中，默认放在/usr/local/mysqldata中，每一个表有一个.frm表描述文件，放在mysqldata下的具体数据库目录；
  • 独占表空间
    • 在mysqldata/yourdatabase下，每一个表都有一个.frm表描述文件，还有一个.ibd，每个表的数据和索引单独存放在以表命名的ibd文件中；

大型网站架构演化

发表于 2019-08-31 | 更新于 2020-02-08 | 分类于 架构
本文字数： 2.3k | 阅读时长 ≈ 2 分钟
  • 本文为《大型网站技术架构：核心原理与案例分析》第一章的读书笔记总结
  • 第一阶段：
    • 文件，应用，数据库
    • LAMP
    • 文件，应用程序和数据库共存于一台服务器中
  • 第二阶段：
    • 文件，应用，数据库
    • 应用服务和数据服务分离
    • 三台服务器：应用服务器，文件服务器和数据库服务器
    • 针对不同需求定制相应的服务器功能
    • 原因
      • 单一服务器不能满足文件容量，抑或是CPU处理的需求
  • 第三阶段：
    • 分布式缓存集群，文件，应用(本地缓存)，数据库
    • 缓存
    • 新增本地缓存和远程分布式缓存集群
    • 原因
      • 根据八二原则：热点数据只有百分之20，所以将其缓存
  • 第四阶段：
    • 负载均衡，应用集群(本地缓存)，文件，分布式缓存集群，数据库
    • 应用服务器集群(负载均衡器)
    • 原因
      • 单一服务器无法处理过量的请求
    • 负载均衡器可以将用户请求分发到任意服务器上
  • 第五阶段：
    • 负载均衡，应用集群(本地缓存，数据访问模块)，分布式缓存集群文件，数据库(主从热备，读写分离)
    • 数据库（主从热备，读写分离）
    • 原因：
      • 缓存之后，读操作从缓存中读取，除了部分缓存访问不命中，缓存过期；全部写操作
      • 以上操作，单一数据库不能很好的满足
  • 第六阶段
    • CDN，反向代理，负载均衡，应用集群(本地缓存，数据访问模块)，分布式缓存集群，文件，数据库(主从热备，读写分离)
    • 反向代理和CDN
      • CDN:网络提供商的机房
      • 反向代理：网站的中心机房(最先访问)
    • 原因：
      • 优化用户体验，提高网站加载速度；
      • 尽早的返回数据给用户，加快用户访问速度，减轻后端服务器的负载压力
  • 第七阶段
    • CDN，反向代理，负载均衡，应用集群(本地缓存，统一数据访问模块)，分布式缓存集群，分布式文件，分布式数据库
    • 分布式数据库，分布式文件服务器
    • 关于分布式数据库：
      • 分布式数据库只有在单表数据规模非常庞大
      • 更常用手段是业务发分库，将不同业务的数据库部署在不同的物理服务器。
    • 原因：数据规模扩大
  • 第八阶段
    • CDN，反向代理，负载均衡，应用集群(本地缓存，统一数据访问模块)，分布式缓存集群，分布式文件，分布式数据库
    • 分布式数据库，分布式文件服务器，搜索引擎，NoSQL
    • NoSQL,搜索引擎
    • 原因：
      • NoSQL,搜索对可伸缩的分布式特性具有更好的支持
      • 应用服务器用过统一数据访问模块访问各种数据，减轻应用程序管理诸多数据源的麻烦
  • 第九阶段
    • CDN，反向代理，负载均衡，应用集群A,消息队列，应用集群B(本地缓存，统一数据访问模块)，分布式缓存集群，分布式文件，分布式数据库
    • 应用集群粒度细化，消息队列
      • 将网站拆分成许多不用的应用，每个应用独立部署维护
      • 通过超链接或者通过消息队列进行数据分发
  • 第10阶段
    • CDN，反向代理，负载均衡，应用集群A,消息队列，应用集群B，分布式服务(本地缓存，统一数据访问模块)，分布式缓存集群，分布式文件，分布式数据库
    • 分布式服务
      • 将可复用的应用提取出来，应用系统只需管理用户界面，通过分布式服务调用公用业务服务

问题分析

根据错误提示，现在只有奇数个URI，即xsi：schemaLocation 前后不匹配

经过对比，发现书中案例的XML标签存在大小写。将Spring改成spring 即可。

原理探索

xml文档解析

  • 这里的xml遵守Schema约束。以下是逐行分析

  • 第一行

说明这是一个xml文件，相当于告诉解析器这是个XML文档，你需要用XML解析器解析

  • 第二行

    xmlns ： XML Namespaces （命名空间）；

    beans 是xml的根节点

    所在行的命名空间，是一种默认命名空间（只能存在一个）。没有使用命名空间前缀，表示未使用其他命名空间的所有标签的默认命名空间，相当于一个兜底。

  • 第三行

    w3c 规范， xsi 对应到XMLSchema-instance

  • 第四行

    具体用到的资源，用于声明目标名称空间的模式文档。

    xsi:schemaLocation属性的值由多个URI引用组成，两个URI之间以空白符分隔。第一个URI是命名空间的名字，第二个URI给出模式文档的位置，模式处理器将从这个位置读取模型文档，该模式文档的目标命名空间必须与第一个URI像匹配

XML文档加载顺序

Spring 在加载xsd文件时总是先试图在本地查找xsd文件(Spring的jar包中已经包含了所有版本的xsd文件)，如果没有找到，才会转向URL指定的路径下载。 Spring加载xsd文件的类是PluggableSchemaResolver,或者通过日志

了解spring 是如何加载xsd文件。此外，Spring 在本地找不到需要的文件向网站下载原因： Spring jar 包的版本低于xsd的文件版本，如果此时发生了断网问题，就会出现问题。

总结（个人意见）

Spring主要是对bean(对象)的处理，所以在配置文件中把beans作为默认命名空间。又因为Schema约束（w3），所以xmln需要与url对应。问题就出在是否遵守xml文件的约束和解析条件，以便xml文件可以找到http://www.springframework.org/schema/beans 与http://www.springframework.org/schema/beans/spring-beans.xsd。
