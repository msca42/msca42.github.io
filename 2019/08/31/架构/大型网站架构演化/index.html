大型网站架构演化

发表于 2019-08-31 | 更新于 2020-02-08 | 分类于 架构
本文字数： 2.3k | 阅读时长 ≈ 2 分钟
  • 本文为《大型网站技术架构：核心原理与案例分析》第一章的读书笔记总结
  • 第一阶段：
    • 文件，应用，数据库
    • LAMP
    • 文件，应用程序和数据库共存于一台服务器中
  • 第二阶段：
    • 文件，应用，数据库
    • 应用服务和数据服务分离
    • 三台服务器：应用服务器，文件服务器和数据库服务器
    • 针对不同需求定制相应的服务器功能
    • 原因
      • 单一服务器不能满足文件容量，抑或是CPU处理的需求
  • 第三阶段：
    • 分布式缓存集群，文件，应用(本地缓存)，数据库
    • 缓存
    • 新增本地缓存和远程分布式缓存集群
    • 原因
      • 根据八二原则：热点数据只有百分之20，所以将其缓存
  • 第四阶段：
    • 负载均衡，应用集群(本地缓存)，文件，分布式缓存集群，数据库
    • 应用服务器集群(负载均衡器)
    • 原因
      • 单一服务器无法处理过量的请求
    • 负载均衡器可以将用户请求分发到任意服务器上
  • 第五阶段：
    • 负载均衡，应用集群(本地缓存，数据访问模块)，分布式缓存集群文件，数据库(主从热备，读写分离)
    • 数据库（主从热备，读写分离）
    • 原因：
      • 缓存之后，读操作从缓存中读取，除了部分缓存访问不命中，缓存过期；全部写操作
      • 以上操作，单一数据库不能很好的满足
  • 第六阶段
    • CDN，反向代理，负载均衡，应用集群(本地缓存，数据访问模块)，分布式缓存集群，文件，数据库(主从热备，读写分离)
    • 反向代理和CDN
      • CDN:网络提供商的机房
      • 反向代理：网站的中心机房(最先访问)
    • 原因：
      • 优化用户体验，提高网站加载速度；
      • 尽早的返回数据给用户，加快用户访问速度，减轻后端服务器的负载压力
  • 第七阶段
    • CDN，反向代理，负载均衡，应用集群(本地缓存，统一数据访问模块)，分布式缓存集群，分布式文件，分布式数据库
    • 分布式数据库，分布式文件服务器
    • 关于分布式数据库：
      • 分布式数据库只有在单表数据规模非常庞大
      • 更常用手段是业务发分库，将不同业务的数据库部署在不同的物理服务器。
    • 原因：数据规模扩大
  • 第八阶段
    • CDN，反向代理，负载均衡，应用集群(本地缓存，统一数据访问模块)，分布式缓存集群，分布式文件，分布式数据库
    • 分布式数据库，分布式文件服务器，搜索引擎，NoSQL
    • NoSQL,搜索引擎
    • 原因：
      • NoSQL,搜索对可伸缩的分布式特性具有更好的支持
      • 应用服务器用过统一数据访问模块访问各种数据，减轻应用程序管理诸多数据源的麻烦
  • 第九阶段
    • CDN，反向代理，负载均衡，应用集群A,消息队列，应用集群B(本地缓存，统一数据访问模块)，分布式缓存集群，分布式文件，分布式数据库
    • 应用集群粒度细化，消息队列
      • 将网站拆分成许多不用的应用，每个应用独立部署维护
      • 通过超链接或者通过消息队列进行数据分发
  • 第10阶段
    • CDN，反向代理，负载均衡，应用集群A,消息队列，应用集群B，分布式服务(本地缓存，统一数据访问模块)，分布式缓存集群，分布式文件，分布式数据库
    • 分布式服务
      • 将可复用的应用提取出来，应用系统只需管理用户界面，通过分布式服务调用公用业务服务
  • 参考来源：《大型网站技术架构：核心原理与案例分析》
