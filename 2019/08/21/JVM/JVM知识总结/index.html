正文

JVM是什么？

  • Java Virtual Machine，java虚拟机，主要任务是编译字节码文件，生成机器码。
  • Sun的定义有所区别:JVM指Java HotSpot Client and Server VM和Interpreter
  • Java的平台无关性的重要体现就是JVM；因为JVM处理的对象字节码是跨平台的（write once ,run everywhere），而且JVM屏蔽了操作系统软硬件的差异，所以只要在不同操作系统上搭建对应的JVM平台就可以完成相应的编译过程。

JVM组成?

  • JVM主要由类加载器，运行时数据区，执行引擎，本地接口组成

编译与优化

JVM处理的对象是class文件，它是通过java文件javac编译而来的，然后通过JVM生成机器码，亦或是直接将java文件编译称本地机器码。上述过程涉及了java的几种编译器。

  • 前端编译器：Sun的Javac。
  • JIT编译器：HotSpot VM的C1，C2编译器。
  • AOT编译器：jaotc。
javac的编译过程
  • 解析和填充符号表过程
    • 词法分析（parseFile）
      • 将源代码的字符流转变为标记（Token）集合;
      • 单个字符式程序编写过程的最小元素，标记是编译过程的最小元素。
      • 标记范围：关键字，变量名，字面量，运算符
    • 语法分析
      • 根据Token序列构造抽象语法树（AST）
      • AST是一种用来描述程序代码语法结构的树形表示方式，每个节点都代表着程序代码中一个语法结构
    • 输入到符号表(enterTrees)
      • 符号表是由一组符号地址和符号信息构成的表格。
      • 如果用户代码中没有提供任何构造器，填充时自动加入默认构造函数
      • 符号表的作用
        • 语义分析:语义检查
        • 目标代码生成阶段，对符号名进行地址分配，符号表是地址分配依据
  • 插入式注解处理器的注解处理过程
  • 语义分析与字节码生成过程
    • 标注检查(attribute)
      • 变量使用前是否已声明
      • 变量与赋值之间的数据类型是否匹配
      • 常量折叠
        • 常量优化机制
        • 举例说明：语法树 int a=1+2;编译期间 int a=3;
    • 数据流分析(flow)
      • 对程序上下文逻辑更进一步的验证
      • 程序局部变量在使用前是否由赋值
      • 方法的每条路径是否都有返回值
      • 是否所以的受查异常都被正确处理
    • 解语法糖（desugar）
      • 将java语法糖还原回简单的基础语法结构
    • 生成字节码
      • 将前面生成的信息转化称字节码写到磁盘中
      • 添加实例构造器()和类构造器()
        到语法树中
      • 这里生成构造器的过程是一个收敛的过程：
        • 先执行父类的实例构造器，然后初始化变量，最终执行语句块
      • 优化：字符串的加操作替换为StringBuffer或StringBuilder（jdk>1.5）的append()
      • 由ClassWriter类的writeClass()方法输出字节码，生成最终的Class文件

运行期优化

解释器和编译器

  • HotSpot同时包含解释器（interpreter）和编译器(c1,c2)
    采用混合模式解释器+一个编译器
  • 解释器
    • 程序迅速启动和执行的时候
    • 内存资源限制,采用解释器节约内存
    • 作为逃生门，逆优化，提高效率
  • 编译器（即时编译）
    • 将代码编译成本地代码，提高执行效率
    • C1:Client Compiler
    • C2:Server Compiler

分层策略

  • 第0层:程序解释执行，解释器不开启性能监控（Profiling）,可触发第一层编译
  • 第1层：C1编译，将字节码编译为本地代码，进行简单，可靠的优化，如有必要将加入新跟那个监控的逻辑
  • 第2层(及以上)：C2编译，将字节码编译为本地代码，但是会启用一些编译耗时较长的优化，甚至会根据性能监控信息进行一些不可靠的激进优化
  • 分层之后，C1,C2同时工作
    • C1获取更高的编译速度
    • C2获取更好的编译质量
    • 解释执行不需要收集性能监控信息

热点代码

  • 热点探测（Hot Spot Detection）
    • 基于采样的热点探测
      • 周期性检查线程的栈顶，经常出现即可
      • 优点：实现简单，高效，很容易地获取方法调用关系
      • 缺点：易受线程阻塞
    • 基于计数器的热点探测 （HotSpot）
      • 为每个方法（甚至代码块）建立计数器
  • 被多次调用的方法
    • 以整个方法作为编译对象，标准的JIT编译方式
  • 被多次执行的循环体
    • 以整个方法为编译对象，栈上替换(OSR编译)

HotSpot为每个方法提供了两个计数器：

  • 方法调用计数器（Invocation Counter）
  • 回边(Back Edge Counter)
  • 超过阈值，触发JIT
    • 方法调用计数器 Client:1500 Server:10000
    • 一个方法被调用
      • 该方法是否存在被JIT编译过的版本
      • 存在，优先使用本地代码
      • 不存在，此方法的调用计数器值加1，判断方法调用计数器+回边计数器>方法调用阈值，向JIT提交该方法的代码编译请求
      • 执行引擎继续进入解释器，循环之前的流程，直到编译完成。
      • 如果一段时间，方法调用计数未满足要求，进入半衰期，热度衰减
    • 回边计数器
      • 统计一个方法中循环体代码的执行的次数
      • 回边：在字节码中遇到控制流向后跳转的指令
      • 目的：触发OSR编译
